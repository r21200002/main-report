    <div class="section">
        <h2>要点</h2>
        <h3>1. 現在の業務作業内容に関する仮説説明</h3>
        <ul>
            <li>理科大では現在、リコーのデバイスマネージャーで印刷ジョブのログを集計し、弊社のジョブログ集計ツールを通している。</li>
            <li>このツールから、「リコー提出用ログファイル（日付別）」と「学内請求用データ（ユーザー部門別）」の2種類のCSVファイルが出力されていると想定されている。</li>
            <li>学内請求用ファイルはExcelマクロを通して、最終的に各部署・各組織向けの請求明細ファイルになっていると仮定されている。</li>
        </ul>

        <h3>2. カンタンセキュアアプリ（リコーのクラウド印刷システム「EDWクラウド」）導入時の経費振り替え作業パターン</h3>
        <ul>
            <li><strong>パターン1: 機器ログ機能の利用</strong>
                <ul>
                    <li>現在の作業部分から変更がないとされており、EDWクラウドから出力されるログは現状のデバイスマネージャーのログと類似した内容（印刷ジョブの詳細情報）を含んでいる。</li>
                    <li>出力されたログはジョブログ集計ツールを通し、その後の流れは現状と同じ。</li>
                </ul>
            </li>
            <li><strong>パターン2: レポート機能の利用</strong>
                <ul>
                    <li><strong>リコー提出用レポート（機器別）：</strong> EDWクラウドの管理サイトで月に一度事前に設定し、レポートを予約することで、機器別の集計レポートを自動で作成することが可能。このレポートはリコーに提出される想定。</li>
                    <li><strong>学内請求用レポート（分類コード別）：</strong> お客様の各組織・各部署に対して、印刷情報経費等をまとめて請求するためのレポートで、レポート機能の中の分類コード別レポートを使用する想定。</li>
                    <li>事前準備として、部門マスターの作成（部門とユーザーの紐付け）、EDWクラウドの管理サイトでの分類コードの使用設定と登録、部門マスターと紐付けた形でのレポート予約が必要。</li>
                    <li>部門コードは各部署・各組織の紐付けを行う想定だが、組織再編時にはメンテナンスが必要となる。</li>
                    <li>出力された請求レポートの情報をもとに、お客様側で手動で金額などを把握し、最終的な請求明細（Excelファイル）を作成すると仮定されている。</li>
                    <li>部門マスターは、ユーザーIDと部門コードを紐付けるファイル形式。</li>
                    <li>分類コード別レポートでは分類コードと部門コードの両方が表示可能だが、部門コード別レポートでは分類コードは表示されない。</li>
                </ul>
            </li>
        </ul>

        <h3>3. 議論の内容</h3>
        <ul>
            <li><strong>クラウド印刷システム導入のメリット:</strong>
                <ul>
                    <li>現在の業務内容のほとんどがクラウド印刷システムでも実現可能であると説明された。</li>
                    <li>クラウド化のメリットとして、オンプレミスサーバーのOS更新やパッチ適用作業（土日作業や有償対応）が不要になる点、およびパッチ適用によるシステム不具合のリスクがなくなるため、安心安全に運用できる点が強調された。</li>
                </ul>
            </li>
            <li><strong>部門マスターのメンテナンスとレポート機能:</strong>
                <ul>
                    <li>組織変更があった際の部門マスターのメンテナンスについて、レポート出力時に最新化されていれば良いことが確認された。</li>
                    <li>集計時に部門マスターに登録されていないユーザーがいた場合、そのユーザーを調べて修正し、再集計が可能であることが確認された。</li>
                    <li>人事異動のタイミング以外に臨時異動などが発生した場合も、集計をかけて所属していないユーザーを調べ、修正・再集計を行う運用が説明された。</li>
                    <li>現在のジョブログ集計ツールでは、部門ユーザー別、機器別、機器ユーザー別の集計が可能であることが確認された。</li>
                    <li>「機器ユーザー別」の集計は、どの部門のユーザーがどの機械から印刷したかを知りたいニーズがあるため、引き続き必要であることが述べられた。</li>
                    <li>現状、リコー提出には「機器別」、学内請求には「部門分類ユーザー別」を使用しており、これに加えて「機器ユーザー別」レポートも必要であると説明された。</li>
                </ul>
            </li>
            <li><strong>データ連携と集計作業に関する課題:</strong>
                <ul>
                    <li>レポート機能で出力されるCSV/Excelフォーマットがジョブログ集計ツールのものと異なる場合、既存のExcelマクロを新しいフォーマットに合わせる修正が必要だが、これはそれほど手間ではないと認識された。</li>
                    <li>お客様（理科大）側から、特定の部門ごとの合計値（部門コードごとの集計）が出力されれば、手動での集計作業を削減できるとの要望があり、リコー側で内部検討が可能であることが示された。</li>
                    <li>現在、手動でのExcel加工（部門ごとのソートや集計）が発生しており、リコー側でその業務代行が可能かを検討する旨が伝えられた。ただし、この代行は「付け替え」のようなお客様の内部事情が発生しない場合に限られる。</li>
                    <li>お客様側は、リコー側でこの作業を代行してもらえるなら助かると述べている。</li>
                    <li>現状の月次データ作成は、毎月10日締めで11日に集計し、翌10日の翌営業日に学内請求データを管財課に渡していることが説明された。管財課向けデータ作成作業は30分ほどかかるとのこと。</li>
                    <li>月次データ公開までにタイムラグがあり、修正が必要な場合に手間がかかるという懸念が示された。</li>
                </ul>
            </li>
            <li><strong>請求関連の課題と要望:</strong>
                <ul>
                    <li>お客様（理科大）から、管財課の要望として、請求を部署ごとに分割して直接請求してほしいという提案があった。これにより管財課の省力化が見込まれるとのこと。</li>
                    <li>リコー側では、これを対応可能か確認が必要であると述べ、請求書を複数発行することになるため、費用面やエビデンスとしての管財課への提出方法なども検討が必要であるとした。</li>
                </ul>
            </li>
            <li><strong>分類コードによる運用改善案:</strong>
                <ul>
                    <li>部門マスターのメンテナンスを不要にするアイデアとして、現在特定の機械で使われている分類コードを「全体分類コード」とし、印刷時にユーザーが分類コードを選ぶようにすれば、部門コードのメンテナンスが不要になる可能性が提案された。</li>
                    <li>これにより、リコーが部署ごとに直接請求することも可能になり、情報システム課は部門メンテナンスが不要になり、管財課も確認作業が省力化されると説明された。</li>
                    <li>ただし、分類コードだけの運用の場合、ユーザー情報と紐付けしないため、異常な印刷量があった際にどのユーザーが使用したか特定しにくくなる可能性があるという懸念も示された。</li>
                    <li>現状、分類コードを使っている部署と使っていない部署があり、複数の分類コードを1つの部署で合算処理しているケースもあるとのこと。</li>
                </ul>
            </li>
            <li><strong>クラウド移行の動機と費用:</strong>
                <ul>
                    <li>クラウド移行によって作業工数が減らない場合でも、サーバーOSの保守やパッチ適用作業の解消といったシステム的なメリットを享受したいのか、それとも現状維持が良いのか、理科大の方針が確認された。</li>
                    <li>お客様（理科大）は、更新するならOS保守などの費用がなくなるのは良いと述べ、現状の運用をできるだけ変えないようにしたい意向を示した。</li>
                    <li>お客様（理科大）は、作業代行サービスを含めた費用見積もりを希望しており、それを含めて社内で検討を開始したいと述べた。</li>
                    <li>リコー側はすでにキャンパスの設置場所を確認していることに言及があったが、リコー側は契約に含まれない古い機械の稼働など、設置場所を含めた現状把握の必要性を述べた。</li>
                </ul>
            </li>
        </ul>

        <h3>4. 決定事項</h3>
        <ul>
            <li>現在の業務作業内容の仮説は合意された。</li>
            <li>リコーのクラウド印刷システム（カンセキアプリ）導入時の経費振り替え作業について、機器ログ機能とレポート機能の2つのパターンで検討を進めることになった。</li>
            <li>部門マスターのメンテナンスは組織変更のたびに必要であり、再集計が可能であることは確認された。</li>
            <li>リコー側で、部門ごとの合計値が出力されるレポートの提供および業務代行（Excel加工による集計作業）の可否を検討することになった。</li>
            <li>クラウド化によってOS保守などの作業がなくなるメリットを考慮しつつ、現状の運用をできるだけ変えない方向で進めることをお客様は希望している。</li>
            <li>リコー側で、作業代行を含めた費用見積もりを提示し、理科大側で社内調整を進めることになった。</li>
        </ul>
    </div>

    <div class="section">
        <h2>実施項目</h2>
        <h3>リコージャパン側の今後の予定</h3>
        <ol>
            <li>機器別レポートに関する具体的な処理がお客様（理科大）側で可能かどうか、社内で現在確認を進める。</li>
            <li>お客様（理科大）が現在行っている部門ごとの集計作業の代行が可能かどうかを内部で検討する。これには、簡単セキュアプリントのツールで出力されたデータで理科大の既存フォームが実現可能かの検証と、それに伴う工数と原価の確認が含まれる。</li>
            <li>管財課からの要望である「部署ごとの直接請求」の可否についても確認を進める。</li>
            <li>作業代行を含めた費用感に関する回答を夏休み明けを目途に提示できるよう努める。</li>
            <li>費用面を含め、情シスと管財課の打ち合わせが始まる前に、すべての要望が盛り込まれた状態での費用感を提示したいと考えている。</li>
            <li>状況について逐一お客様に共有する。</li>
        </ol>

        <h3>東京理科大学側の今後の予定</h3>
        <ol>
            <li>現在の請求書発行の年間スケジュールを考慮し、リコーからのデータが月末（25日前後）までに部門に公表されるよう、11日の締めから25日までの期間で調整可能か検討する。</li>
            <li>クラウド移行で作業工数が減らなくても、システム的なメリット（OS保守などの削減）を重視するか、または現状維持を希望するかを内部で検討し、リコーに伝えることになる。</li>
            <li>提供された機器リストにないが稼働している古い機械（個別契約）の存在についても、設置場所を含めて洗い出す必要があるかもしれない。</li>
            <li>部門マスターのメンテナンスを不要にする「分類コード全体での運用」のアイデアについても検討を進める。</li>
        </ol>
    </div>
